Input Files

Input files are used to determine the configuration and set of experiments to run when calling out to the system to run experiments. The goal of these files is to allow users to specify a list of experiments to be run, where each experiment has an Application with a set of parameters relevant to it, such as weighting method for influence maximization, and each experiment also has an Algorithm with a set of relevant parameters, such as the time horizon. In addition, there is a set of global parameters, used when running the experiment, and a copy of these global parameters are passed to every experiment in case they are needed for something.

In order to reduce the amount of writing needed, both the globals and each application and algorithm have a set of default values for all parameters coded in a parameter file. Users only need to specify the values for parameters whose value differs from the default.
Additionally, there may be cases where a user wants to use the same Application or Algorithm with the same parameters in multiple different experiments. To avoid rewriting the same Application or Algorithm specification multiple times, experiments are specified by a list field called experiments where each entry gives the index of the Application and Algorithm specification it wants used.

Input files are written as JSON files. (See here for a reference on JSON files.)

For information on generating input files for combinations of parameters, rather than producing them manually, see here.

fields:
  • global_vars:
    • type: object
    • description:

      This is used to specify the value of any global parameters that should not match the default values.
      An explanation of the global parameters and their default values can be found here.

    • required: false
    • default: Empty dictionary. Causes the default value for each global parameter, as specified in the global parameter file, to be used.
  • unique_globals:
    • type: boolean
    • description:

      In addition to the system running the experiments, each experiment configuration in the generated list will receive the global parameters, in case an experiment needs access to them for some reason. If this variable is set to true, every experiment configuration will receive a unique copy of the object specifying the globals, if it is false, every experiment configuration will receive the same object reference.

    • required: false
    • default: false
  • applications:
    • type: list
    • description:

      This is a list of application specifications (application and parameters to be used) to be used in the experiments.

    • required: true
    • minLength: 1
    • items:
      • type: object
      • description:

        The specification for an application setup to be used by one or more experiments.

      • fields:
        • name:
          • type: string
          • description:

            The name of the application.

          • required: true
        • unique:
          • type: boolean
          • description:

            An individual application specification can be paired with more than one algorithm specification in the experiments to be run. This parameter dictates whether each experiment setup that the application specification is used in receives its own unique copy of the parameters being passed in, in order to guard against other instances making changes, or if it will use the same object of parameters to save memory usage.

            If true, each experiment setup will receive its own unique copy of the parameters. Note that this is not the same thing as each run of an experiment setup receiving its own unique copy of the parameters.

          • required: false
          • default: The default value used is whatever value is specified for the variable default_unique_params found in the file Code/Utilities/program_vars.py.

            At the time of writing, that value is false.
        • unique_instance:
          • type: boolean
          • description:

            An individual application specification can be paired with more than one algorithm specification in the experiments to be run. This parameter helps dictate whether each experiment setup that the application specification is used in receives its own unique instance of the Application being used, in order to guard against other instances making changes to some internal state, or if it will use the same object to save memory usage.

            If true, each experiment setup will receive its own unique instance of the Application. Note that this is not the same thing as each run of an experiment setup receiving its own unique instance of the Application.

            If false, each experiment setup that uses this application specification will use the same Application instance unless each Application instance needs to have a unique copy of the parameters.

          • required: false
          • default: The default value used is the logical OR of the value of the unique field and whatever value is specified for the variable default_unique_instance found in the file Code/Utilities/program_vars.py.

            At the time of writing, that value is false.
        • params:
          • type: object
          • description:

            Object specifying the values to use for application parameters when running experiments. Any parameters not specified in this object that are not calculated using other parameters will take on the default values specified in the parameter file for the application.

          • required: false
          • default: The default value is an empty object, meaning that every parameter will take on the default value specified in the parameter file.
          • fields:
            • r'.+':
              • description:

                This is a pattern property. The name is not literal (or even the correct regular expression, because no one regular expression could match every valid Python identifier), it is meant to match the name of any parameter written in the application's parameter file. The value is whatever the user wishes that parameter value to be, provided it will pass the validation in the parameter file.
                There can be more than one of these.

              • type: Any
              • required: False
              • default: The parameter will take on its default value as specified in the parameter file.
  • algorithms:
    • type: list
    • description:

      This is a list of algorithm specifications (algorithm and parameters to be used) to be used in the experiments.

    • required: true
    • minLength: 1
    • items:
      • type: object
      • description:

        The specification for an algorithm setup to be used by one or more experiments.

      • fields:

          • type: string
          • description:

          • type: object
          • description:

            Object specifying the values to use for algorithm parameters when running experiments. Any parameters not specified in this object that are not calculated using other parameters will take on the default values specified in the parameter file for the application.

          • required: false

  • experiments:
    • type: list
    • description:

      A list of which application specifications and algorithm specifications should be paired together and run as experiments.

    • required: true
    • items:
      • type: object
      • fields:
        • application:
          • type: integer
          • description:

            The index within the applications list of the specification for the application to use for this experiment.

          • minimum: 0
          • maximum: applications.length - 1
          • required: true
        • algorithm:
          • type: integer
          • description:

            The index within the algorithms list of the specification for the algorithm to use for this experiment.

          • minimum: 0
          • maximum: algorithms.length - 1
          • required: true

Files for Geneating Input Files

Input files are used to determine the configuration and set of experiments to run when calling out to the system to run experiments. However, sometimes users want to specify a list of values that a particular parameter can take on and then run tests where all possible combinations of parameters are tried. The purpose of these files is to allow users to specify Application and Algorithm specifications where each parameter specified is a list of values that the would like to use in combination with other sets of parameters in the specification. For each specification, all possible combinations of parameter values are calculated. From those expanded lists of specifications, all possible combinations of Application and Algorithm specifications are taken, tested for validity, and those that are valid are made into an input file that can be passed to the system to be run.

In order to reduce the amount of writing needed, both the globals and each application and algorithm have a set of default values for all parameters coded in a parameter file. Users only need to specify the sets of possible parameter values for parameters where they would like to test more than one possible value or the value they would like to test differs from the default.
Note: The combination effect is not applicable to globals.

Files to generate input files are written as JSON files. (See here for a reference on JSON files.)

fields:
  • global_vars:
    • type: object
    • description:

      This is used to specify the value of any global parameters that should not match the default values.
      An explanation of the global parameters and their default values can be found here.

    • required: false
    • default: Empty dictionary. Causes the default value for each global parameter, as specified in the global parameter file, to be used.
  • unique_globals:
    • type: boolean
    • description:

      In addition to the system running the experiments, each experiment configuration in the generated list will receive the global parameters, in case an experiment needs access to them for some reason. If this variable is set to true, every experiment configuration will receive a unique copy of the object specifying the globals, if it is false, every experiment configuration will receive the same object reference.

    • required: false
    • default: false
  • applications:
    • type: list
    • description:

      This is a list of application specifications (application and parameters to be used) to be used in the experiments.

    • required: true
    • minLength: 1
    • items:
      • type: object
      • description:

        The specification for an application setup to be used by one or more experiments.

      • fields:
        • name:
          • type: string
          • description:

            The name of the application.

          • required: true
        • unique:
          • type: boolean
          • description:

            An individual application specification can be paired with more than one algorithm specification in the experiments to be run. This parameter dictates whether each experiment setup that the application specification is used in receives its own unique copy of the parameters being passed in, in order to guard against other instances making changes, or if it will use the same object of parameters to save memory usage.

            If true, each experiment setup will receive its own unique copy of the parameters. Note that this is not the same thing as each run of an experiment setup receiving its own unique copy of the parameters.

          • required: false
          • default: The default value used is whatever value is specified for the variable default_unique_params found in the file Code/Utilities/program_vars.py.

            At the time of writing, that value is false.
        • params:
          • type: object
          • description:

            Object specifying the parameters for which values other than the default are to be tested. Corresponding to each provided parameter must be a list of values for that parameter to take on. The sets of values from each list will be exaustively combined to form a list of application specifications for each individual applications, those lists of application specifications will be combined into a single list of application specifications which will then be exhaustively combined with the generated list of algorithm specifications to form a complete list of experiments.

          • required: false
          • default: The default value is an empty object, meaning that every parameter will take on the default value specified in the parameter file.
          • fields:
            • r'.+':
              • description:

                This is a pattern property. The name is not literal (or even the correct regular expression, because no one regular expression could match every valid Python identifier), it is meant to match the name of any parameter written in the application's parameter file. The value is a list containing whatever values the user wishes that parameter value to be tested with, provided they will pass the validation in the parameter file.
                There can be more than one of these.

              • type: list
              • minLength: 1
              • required: False
              • default: List with only the parameter's default value in it. Meaning that the parameter will take on its default value as specified in the parameter file.
              • items:
                • type: Any
                • description:

                  The value is whatever the user wishes that parameter value to be, provided it will pass the validation in the parameter file.

    • description:

    • minLength: 1

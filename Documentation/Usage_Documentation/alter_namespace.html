alter_namespace.py

alter_namespace.py is a Python file that serves as an interface to enable the user to change the configuration of parts of the namespace. It also enables the user to obtain information about the current configuration of parts of the namespace, to register and unregister algorithms and applications from the namespace system's list of applications and algorithms, as well as to see the lists of currently registered applications and algorithms.

Running alter_namespace.py without any arguments will cause usage information to be printed to the console.

alter_namespace.py supports several commands, each of which has its own set of accepted and required arguments. Each command is identified by both a number and a name.
To run a command, run alter_namespace.py and supply the first argument as '-' followed by either of the command's identifiers. The remaining arguments should be any arguments you wish to supply for the command you are calling. Arguments passed to commands can appear in any order.
Example: The invert command is identified by both the name invert and the number 0. Additionally, the only argument it accepts is optional. If we were to run the command without any additional arguments, the arguments we pass when running alter_namespace.py could be either of:

  • -0
  • -invert
All together, when running from the command line (using python3 in the examples), this would be:
  • python3 alter_namespace.py -0
  • python3 alter_namespace.py -invert

IMPORTANT NOTE: The changes performed on the configuration of file paths by using this interface are committed to version control. This feature is provided in order to enable users to easily clone the project and run experiments on remote machines and computing clusters. However, if multiple users have clones of the project with their own result files, changes that one user makes to the configuration and pushes will be copied the next time the other user performs a git pull, but changes to the file system resulting from the configuration changes will not be copied. To work around this, users should either coordinate when they change configurations, or use separate branches for running experiments.

Commands

  1. invert

    This command changes the order in which the path parts for applications and algorithms appear.

    Arguments

    • (optional) app or 1 to indicate that applications should go first, alg or 2 to indicate that algorithms should go first.
      This argument is used to indicate which of the application and algorithm parts should appear first, rather than just flipping them. Use if you are unsure which is currently appearing first in file paths.

  2. timestamp

    This command changes whether or not a timestamp is used in the joint section of the file path for a given application-algorithm pair.

    Arguments

    • (required) --app=<value>: This argument specifies the application part of the application-algorithm pair for which timestamp usage should be toggled.
      <value> should be replaced by an application name or -all. Using -all indicates that the timestamp toggle operation should be performed for every pairing created by matching all registered applications with the value(s) indicated by the --alg= argument.
    • (required) --alg=<value>: This argument specifies the algorithm part of the application-algorithm pair for which timestamp usage should be toggled.
      <value> should be replaced by an algorithm name or -all. Using -all indicates that the timestamp toggle operation should be performed for every pairing created by matching all registered algorithms with the value(s) indicated by the --app= argument.
    • (required) --default=<value>: This argument serves a different purpose, depending on whether a timestamp is being added or removed.
      If a timestamp is being added, <value> is the timestamp that is assigned to already existing files that do not yet have a timestamp.
      If a timestamp is being removed, of the currently existing experiment result files, only those whose timestamp matches <value> (if any) will be kept. The rest will be deleted.
      <value> should be replaced by a timestamp with second level precision.
    • (optional) --fixed=<value>: This argument is used to specify that timestamp usage should only be turned on or only be turned off, as opposed to being toggled. This is useful when setting timestamp settings for multiple application-algorithm pairs at once, where some may have different values for their current setting.
      Valid values are true to use timestamps or false to not use timestamps.

  3. randId

    This command changes whether or not a random ID is used in the joint section of the file path for a given application-algorithm pair.

    Arguments

    • (required) --app=<value>: This argument specifies the application part of the application-algorithm pair for which random ID usage should be toggled.
      <value> should be replaced by an application name or -all. Using -all indicates that the random ID toggle operation should be performed for every pairing created by matching all registered applications with the value(s) indicated by the --alg= argument.
    • (required) --alg=<value>: This argument specifies the algorithm part of the application-algorithm pair for which random ID usage should be toggled.
      <value> should be replaced by an algorithm name or -all. Using -all indicates that the random ID toggle operation should be performed for every pairing created by matching all registered algorithms with the value(s) indicated by the --app= argument.
    • (required) --default=<value>: This argument serves a different purpose, depending on whether a random ID is being added or removed.
      If a random ID is being added, <value> is the random ID that is assigned to already existing files that do not yet have a random ID.
      If a random ID is being removed, of the currently existing experiment result files, only those whose random ID matches <value> (if any) will be kept. The rest will be deleted.
      <value> should be replaced by an eight digit random ID.
    • (optional) --fixed=<value>: This argument is used to specify that random ID usage should only be turned on or only be turned off, as opposed to being toggled. This is useful when setting random ID settings for multiple application-algorithm pairs at once, where some may have different values for their current setting.
      Valid values are true to use random IDs or false to not use random IDs.

  4. joint-break

    This command changes whether or not a path separator or a basic separator is used at the end of the joint section of the file path for a given application-algorithm pair. Note that this setting only takes effect if at least one of a timestamp and a random ID is in use by the pair.

    Arguments

    • (required) --app=<value>: This argument specifies the application part of the application-algorithm pair for which path separator usage should be toggled.
      <value> should be replaced by an application name or -all. Using -all indicates that the path separator toggle operation should be performed for every pairing created by matching all registered applications with the value(s) indicated by the --alg= argument.
    • (required) --alg=<value>: This argument specifies the algorithm part of the application-algorithm pair for which path separator usage should be toggled.
      <value> should be replaced by an algorithm name or -all. Using -all indicates that the path separator toggle operation should be performed for every pairing created by matching all registered algorithms with the value(s) indicated by the --app= argument.
    • (optional) --fixed=<value>: This argument is used to specify that path separator usage should only be turned on or only be turned off, as opposed to being toggled. This is useful when setting path separator settings for multiple application-algorithm pairs at once, where some may have different values for their current setting.
      Valid values are true to use path separators or false to use basic separators.

  5. swap

    For an indicated application or algorithm, this command swaps the positions of two of the parameters within that application/algorithm's namespace configuration. Bear in mind that whether it is a path separator or a basic separator that follows a parameter is tied to the parameter, not the position. So swapping two parameters may change the location in which the path is broken into directory and content.

    Arguments

    Special Requirements:
    Exactly one (no more or less) of the first and second arguments listed is expected.
    This command requires that two instances of the third argument listed be provided: one for each of the swap positions.

    • (semi-required) --app=<value>: This argument indicates that the parameter swap is being performed on an application with the name <value>.
      <value> should be an application name.
    • (semi-required) --alg=<value>: This argument indicates that the parameter swap is being performed on an algorithm with the name <value>.
      <value> should be an algorithm name.
    • (required x2) One of the parameters to be swapped. This can either be the parameter name, or the index of the parameter within the current configuration. Note that the header identifying the application or algorithm is always first in the path section (index 0). As a result, the first parameter index is 1.

  6. break

    For an indicated application or algorithm, changes whether it is a file path separator or a basic separator that follows after the indicated parameter. Note that the presence of a timestamp or random ID will overwrite the choice for the last parameter of whatever set of parameters appears second and simply use a file path separator between that parameter and the timestamp and/or random ID.

    Arguments

    Special Requirements:
    Exactly one (no more or less) of the first and second arguments listed is expected.
    The third argument is also required.

    • (semi-required) --app=<value>: This argument indicates that the path separator (path break) toggle operation is being performed on an application with the name <value>.
      <value> should be an application name.
    • (semi-required) --alg=<value>: This argument indicates that the path separator (path break) toggle operation is being performed on an algorithm with the name <value>.
      <value> should be an algorithm name.
    • (required) --at=<value>: This argument indicates the parameter/header that the addition or removal of a path separator should follow.
      <value> should either be the name of the parameter or the index at which it appears in the path segment. For the header that identifies the application or algorithm, you can either use the index (0) or the special token <START>.
    • (optional) --fixed=<value>: This argument is used to specify that path separator usage should only be turned on or only be turned off, as opposed to being toggled. Use if you are unsure which separator type is currently being used.
      Valid values are true to use path separators or false to use basic separators.

  7. add

    Inserts a given parameter into paths for the indicated application or algorithm at/directly in front of the indicated index/parameter.

    Arguments

    Special Requirements:
    Exactly one (no more or less) of the first and second arguments listed is expected.
    Every remaining argument aside from the last one is required.

    • (semi-required) --app=<value>: This argument indicates that the given parameter should be inserted into the path for an application with the name <value>.
      <value> should be an application name.
    • (semi-required) --alg=<value>: This argument indicates that the given parameter should be inserted into the path for an algorithm with the name <value>.
      <value> should be an algorithm name.
    • (required) --at=<value>: The parameter/position at/in front of which the given parameter should be inserted.
      <value> should either be the name of the parameter in front of which the given parameter should be inserted or the numerical index at which the parameter should be inserted. If it is being added to the end, either the numerical index or the special value <END> can be used. Note that, since the header identifying the application/algorithm always appears first (index 0), the first valid index is 1.
    • (required) The name of the parameter to be added to the path. An error will occur if this parameter does not exist in the parameter file for the specified application/algorithm.
    • (required) --default=<value>: This argument is the parameter value that is assigned to the position the parameter is being added to in already existing files.
      <value> should be replaced by a valid value for the parameter that is appropriate to the experiment setup prior to adding the parameter to the path.
    • (optional) --sep=<value>: This argument is used to indicate whether a path separator or a basic separator should follow the inserted parameter. Valid values are true to use path separators or false to use basic separators.

  8. remove

    Removes a given parameter from the paths for the indicated application or algorithm. The parameter to remove can be identified using either name or numerical index.

    Arguments

    Special Requirements:
    Exactly one (no more or less) of the first and second arguments listed is expected.
    Every remaining argument is also required.

    • (semi-required) --app=<value>: This argument indicates that the given parameter should be removed from the path for an application with the name <value>.
      <value> should be an application name.
    • (semi-required) --alg=<value>: This argument indicates that the given parameter should be removed from the path for an algorithm with the name <value>.
      <value> should be an algorithm name.
    • (required) The name or the index of the parameter to remove.
      Note that, since the header identifying the application/algorithm always appears first (index 0), the first valid index is 1.
    • (required) --default=<value>: This argument is used to specify which files, out of those whose paths are affected by the change, should be kept. Files for whom the current value at the path position of the parameter to be removed matches the given default value will be kept.
      For example:
      If a path segment prototype is alg-etcg/<seed_set_size>/<time_horizon>/
      Now if the following are both path segments of existing files:
      • alg-etcg/10/10000/
      • alg-etcg/10/20000/
      If you are removing time_horizon and the following argument is given: --default=10000 then the first file will be kept and the path segment will become alg-etcg/10/
      The second file will be discarded.

  9. info

    This command produces information on the current namespace configuration one or more applications, algorithms, or a joint sections for one or more application-algorithm pairs.

    Output

    Different formats of output are produced dependent on whether information is being printed on applications/algorithms or on joint sections.

    For each application or algorithm being printed, the format begins with a line containing the application or algorithm name. The next line contains the symbol <START>followed by a space, a / if the separator associated with the first header is a path separator, otherwise a _ will appear to represent a basic separator, this is then followed by another space.
    Then, for every experiment parameter that will appear in the file path (in the order they appear in the path), it will print the name of the parameter along with either / or _, surrounded by spaces.
    After the final separator character and its following space, the symbol <END> will be printed.
    Example: etcg:
    <START> / time_horizon _ seed_set_size / <END>

    For each joint section being printed there will be a line with application_name - algorithm_name:
    The next line will contain one of four different values, depending on the current configuration: Timestamp if the configuration uses only a timestamp, Random ID if the configuration uses only a random id, Timestamp _ Random ID if the configuration uses both a timestamp and a random id, or Nothing if the configuration uses neither a timestamp nor a random id.
    The value that is printed is followed by a space and then / (if the joint section is to be followed by a path separator) or _ (if the joint section is to be followed by a basic separator).
    Example 1:
    im - etcg:
    Timestamp /
    Example 2:
    im - cmab:
    Random ID _
    Example 3:
    im - dart:
    Timestamp _ Random ID _
    Example 4:
    im - csar:
    Nothing /

    Arguments

    Special Requirements:
    At least one of the two listed arguments must be present. If only one is present, information on either applications or on algorithms will be printed. If both are present, it will be information on joint configurations.

    • (semi-required) --app-<value>:
      If this argument is the only one used, it specifies that configuration information should be provided for the application(s) indicated by this argument.

      If the --alg= argument is also provided, then the application(s) indicated by this argument will be exhaustively paired with the algorithm(s) indicated by the --alg= argument and information about the joint configurations of each pair will be provided.

      <value> should either be the name of an application, or -all. If the value is -all, this argument indicates that every application registered in the namespace systemshould be used.
    • (semi-required) --alg-<value>:
      If this argument is the only one used, it specifies that configuration information should be provided for the algorithm(s) indicated by this argument.

      If the --app= argument is also provided, then the algorithm(s) indicated by this argument will be exhaustively paired with the applications(s) indicated by the --app= argument and information about the joint configurations of each pair will be provided.

      <value> should either be the name of an algorithm, or -all. If the value is -all, this argument indicates that every algorithm registered in the namespace systemshould be used.

  10. list

    Lists the names of all applications, algorithms, or both that are registered with the namespace system.

    Arguments

    • (required) This argument is a value to indicate whether registered applications, algorithms, or both should be listed. Valid values are app or 1 to indicate applications, alg or 2 to indicate algorithms, or both or 3 to indicate both.

  11. register

    Adds the indicated application/algorithm to the namespace system's list of all applications/algorithms. These lists are only used by only used by commands with arguments that accept -all as a value. The -all value indicates that the operation should be performed for all applications or algorithms that appear in the corresponding list.

    Arguments

    Special Requirements:
    Exactly one (no more or less) of the arguments listed is expected.

    • (semi-required) --app=<value>: This argument indicates that <value> should be added to the list of all applications.
      <value> should be an application name.
    • (semi-required) --alg=<value>: This argument indicates that <value> should be added to the list of all algorithms.
      <value> should be an algorithm name.

  12. unregister

    Removes the indicated application/algorithm from the namespace system's list of all applications/algorithms. These lists are only used by only used by commands with arguments that accept -all as a value. The -all value indicates that the operation should be performed for all applications or algorithms that appear in the corresponding list.

    Arguments

    Special Requirements:
    Exactly one (no more or less) of the arguments listed is expected.

    • (semi-required) --app=<value>: This argument indicates that <value> should be removed from the list of all applications.
      <value> should be an application name.
    • (semi-required) --alg=<value>: This argument indicates that <value> should be removed from the list of all algorithms.
      <value> should be an algorithm name.
